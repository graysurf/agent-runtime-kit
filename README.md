# agent-runtime-kit

Canonical source tree for local agent runtime content.

This repository owns the files that describe what an agent runtime should load:
skills, plugin metadata, hook sources, policy docs, render templates,
manifests, and validation fixtures. Product-specific runtime homes are rendered
from this source instead of being edited by hand.

## Architecture

`agent-runtime-kit` is organized around one portable source tree plus explicit
product adapters:

- `core/` contains product-independent source content: skills, policies, hooks,
  schemas, helper scripts, and durable design docs.
- `targets/` contains product adapter surfaces. Each target declares how core
  content becomes product-native files, link maps, hook activation, and plugin
  metadata.
- `manifests/` is the machine-checkable inventory for skills, plugins,
  products, runtime roots, and third-party CLI tools.
- `build/` is generated render output. It is regenerated by `agent-runtime
  render` and compared against committed golden snapshots.
- `tests/golden/` pins expected rendered product output.
- `tests/drift/` contains hermetic fixtures for drift-audit behavior.
- `scripts/` contains repository bootstrap and CI helpers.

## Runtime Model

The repo separates source, generated output, live runtime homes, and writable
state:

- `source_root`: this versioned repository.
- `build/`: generated product output for review and golden tests.
- `live_home`: the actual product runtime home managed by install commands.
- `docs_home`: the product-visible documentation root.
- `state_home`: writable runtime state, evidence, temporary output, and backups.
- `plugin_root`: product-specific plugin or skill discovery root.

Runtime state is not tracked in this repository. Generated output should be
reproducible from source content plus manifests.

## CLI Boundary

This repo does not ship its own CLI binary. The `agent-runtime` command lives in
`sympoies/nils-cli` and provides render, install, drift audit, doctor, and
lifecycle commands.

Skills and manifests declare the nils-cli binaries they require through
`required_clis`. Local development may use an unreleased debug build from a
local nils-cli checkout, but released contracts are pinned only after the
nils-cli release and tap update complete.
