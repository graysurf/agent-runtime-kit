# Evidence Archive

The evidence archive makes skill-usage evidence durable and queryable beyond
the ephemeral agent-out runtime tree. It is the durability-and-query lane that
sits beside the curated-lesson lane in
[`heuristic-system/HEURISTIC_SYSTEM.md`](../heuristic-system/HEURISTIC_SYSTEM.md):
the heuristic system retains hand-written lessons; the evidence archive retains
the machine-emitted `skill-usage` records those lessons are drawn from.

## Lifecycle

Evidence moves through five stages. Each stage is owned by one surface; this
policy is the map, not a re-implementation of any of them.

1. **Produce** — skills emit `skill-usage.record.v1` envelopes into the
   agent-out runtime tree (`${AGENT_HOME}/out/projects/<owner__repo>/...`)
   during a session. Each record carries a `producer { tool, nils_cli_version }`
   block so an archived record is attributable to the surface and CLI version
   that wrote it.
2. **Surface** — at session end, `heuristic-session-closeout` enumerates the
   session's records (read-only) and flags non-pass outcomes as promotion
   candidates. This is awareness, not durability: the runtime tree is still
   ephemeral and is reaped on its own retention schedule.
3. **Migrate** — `evidence migrate` (the `evidence-migrate` skill) copies
   records out of the runtime tree into the archive repository. Dry-run is the
   default; `--apply` writes, commits, and pushes. Migration is the durability
   boundary: once a record is archived and pushed, it survives the runtime
   tree's reaping.
4. **Store** — the archive repository holds the rollups, their `metadata.yaml`,
   scrubbed linked evidence, scrub logs, and a derived `catalog.json` (see
   Layout).
5. **Query** — `evidence discover` / `query` / `search` / `catalog` read the
   archive. These are read-only and are driven directly from the CLI.

## Archive repository

The archive is a standalone git repository, resolved (in order) from:

1. `--archive <path>` (explicit),
2. `$AGENT_EVIDENCE_ARCHIVE_HOME`,
3. the machine-local config `$XDG_CONFIG_HOME/agent-evidence-archive/config.yaml`,
4. the XDG data-home default
   `${XDG_DATA_HOME:-$HOME/.local/share}/agent-evidence-archive`.

No surface hardcodes the clone path. The archive is private; it is never
checked into a working repository.

### Layout

```
agent-evidence-archive/
  config/hosts.yaml      # authoritative host classification (schema: agent-evidence-archive.hosts.v1)
  evidence/<host>/<org>/<repo>/...   # archived skill-usage.rollup.v1 + metadata.yaml + scrub log
  catalog.json           # derived index (evidence.catalog.v1); regenerated by migrate --apply / catalog --write
  README.md
```

`config/hosts.yaml` is the authoritative classification of every provider host
whose evidence is archived. Each host declares a `class`:

- `personal` — retained at the maintainer's discretion.
- `employer` — retained only during an active employment relationship with the
  named employer; subject to whole-archive destruction on termination
  (`delete-on-termination`). Add an `employer` host only when a real
  employer-operated host is being archived; no placeholder entries.

A record can only be archived to a host present in `hosts.yaml`. This mirrors
the agent-plan-archive hosts model.

## Identity and host classification

The agent-out `<owner__repo>` directory slug carries no host. `evidence migrate`
resolves the host so a record lands under the correct `evidence/<host>/...`
path:

- It derives identity from the working repo's `cwd` → `origin` remote when the
  record's source can still be located.
- Under a single-host `hosts.yaml`, the sole host is used.
- Under a **multi-host** `hosts.yaml`, a slug-only record is ambiguous. Migrate
  refuses to guess: the record is reported as `blocked` and skipped. The
  operator re-runs with `--host <fqdn>` (a host present in `hosts.yaml`) to
  vouch for the records they recognize. This is a deliberate human decision, not
  an inference.

Blocked records are never fatal. A migrate run continues past every
unresolvable or unreadable record, reports them in `blocked`, and migrates the
rest. An all-blocked run is a successful no-op.

## Redaction discipline

Migration scrubs secrets **before** anything is written to the archive — raw
record bytes never reach the archive. Scrubbing uses the shared `nils-scrub`
pattern set (`REDACTED` token), runs over the rollup, metadata, and any UTF-8
linked-evidence children (binary children are copied verbatim), and writes a
scrub log alongside each record so a reviewer can see what was redacted. The
`evidence-migrate` skill surfaces the per-record scrub summary
(`patterns_triggered` + `total_matches`) in the dry-run for review. Scrubbing is
never disabled. A working directory outside `$HOME` is itself redacted to
`[REDACTED]` rather than archived as an absolute path.

## Idempotency and copy-only semantics

Migration is **copy-only**: the agent-out source tree is left intact (it is
reaped by its own retention, not by migrate). Records dedup by content digest,
so re-running migrate is safe — already-archived records are reported in
`already_archived` and skipped. There is no source-deletion step and no
working-repo push to perform; the CLI commits and pushes only the archive.

## Promotion linkage

A `skill-usage` record can link a heuristic-inbox promotion case
(`promotion.heuristic_inbox_case`). This is the join between the two retention
lanes:

- `evidence migrate --promotion-only` migrates just the records that link a
  case.
- `evidence catalog --case-id <id>` and `evidence query` surface archived
  records by case.

Use this to trace a retained heuristic lesson back to the machine evidence that
motivated it, and vice versa.

## When to migrate

Migrate after a session's goal is achieved and `heuristic-session-closeout` has
surfaced its records, or periodically to drain the runtime tree before it is
reaped. Migration is not a per-task step. Always review the dry-run — counts,
prepared targets, scrub summary, and the `blocked` list — before `--apply`.
