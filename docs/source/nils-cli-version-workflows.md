# nils-cli Version Workflows

## Purpose

This repo pins exactly one `nils-cli` surface (`docs/source/nils-cli-pin.yaml`,
`pinned_tag`) and `scripts/ci/all.sh` Position 2 blocks any push whose host
`agent-runtime` deviates from it — ahead **or** behind. That pin is the consumer
contract: goldens, fixtures, and skill bodies are written for the pinned
surface.

Sometimes you still need the host to run a **different** `nils-cli` version for a
bounded task — without relaxing the gate or moving the pin prematurely:

- **Reproduce a regression** that only an older release exhibits (downgrade).
- **Develop coupled changes** against an unreleased `nils-cli` before it ships,
  in the common loop: open an issue here → change `nils-cli` → return here to
  adjust consumers and test (upgrade-in-progress, pre-bump).
- **Promote the pin** once a release lands (the formal bump).

This document is the source of truth for those three workflows. The mechanics
live in `scripts/dev/with-nils-version.sh`; `DEVELOPMENT.md` keeps the short
recipe and links here. Related sources: `docs/source/nils-cli-pin.yaml` (the
machine-readable pin the gate reads), `docs/source/nils-cli-surface.md` (the
human-readable surface snapshot and consumed binaries), and the
`meta:nils-cli-bump` skill (owns the bump).

## Core principle

> Switching versions temporarily is a host/`PATH` concern. Moving the pin is a
> deliberate act owned by `meta:nils-cli-bump`. **Never edit
> `nils-cli-pin.yaml`, bypass Position 2, or commit goldens regenerated by an
> off-pin binary just to make a coupled-dev branch green.** The pin only moves
> after the target `nils-cli` release ships and the host is upgraded to it.

The consumed surface is the whole set of binaries in
`docs/source/nils-cli-pin.yaml` (`agent-runtime`, `plan-issue`, `plan-tooling`,
`forge-cli`, `plan-archive`, `agent-docs`, …), not just `agent-runtime`. Any
version-switch must put the **full** surface on `PATH`, which is why the helper
prefers a release tarball or a whole-workspace build over a single
`-p agent-runtime-cli` build.

## The version-alignment gate interaction (read this first)

`scripts/ci/all.sh` is a linear stack and Position 2 is the version-alignment
gate. When the host binary is off-pin (any downgrade or unreleased build),
Position 2 **fails and aborts the stack before Positions 3-13 run**. That is by
design — it is the pin check itself.

Consequence: when validating against an off-pin binary you **cannot** run
`all.sh` end to end. Run the **content** gates individually instead — they
exercise the binary's behavior, which is what coupled dev actually needs to
check:

```bash
# Content gates (everything all.sh runs except Position 2), against whatever
# agent-runtime is on PATH:
plan-tooling validate --format text --explain          # Position 1
bash scripts/ci/skill-governance-audit.sh              # Position 1
agent-runtime render --product codex                   # Position 3
agent-runtime render --product claude                  # Position 4
agent-runtime render --target support-matrix           # Position 5
agent-runtime render --product codex  --update-golden >/dev/null
agent-runtime render --product claude --update-golden >/dev/null
agent-runtime render --target support-matrix --update-golden >/dev/null
git diff --exit-code -- tests/golden/                  # Position 6
agent-runtime audit-drift                              # Position 7
bash scripts/ci/sandbox-install-rehearsal.sh           # Position 10
bash tests/runtime-smoke/run.sh --mode deterministic   # Position 11
bash tests/projects/project-local-smoke/run.sh         # Position 12
bash tests/hooks/run.sh                                # Position 13
```

Position 2 is the **last** gate to go green, and it only does so through the
bump (Workflow C), never on a coupled-dev branch.

## Workflow A — Reproduce against an older release (downgrade)

Goal: run a known older released version to confirm or bisect a regression,
without touching Homebrew or the pin.

Preferred — download the released binaries from the GitHub release page (one
tarball, full surface):

```bash
# Run a single command under v0.30.0:
scripts/dev/with-nils-version.sh release:v0.30.0 -- agent-runtime audit-drift

# Or just resolve and inspect the bin dir / version (no command):
scripts/dev/with-nils-version.sh release:v0.30.0
```

If you need source-level inspection of the old code, build from the tag instead:

```bash
scripts/dev/with-nils-version.sh src:v0.30.0 -- plan-tooling validate
```

Neither path mutates the Homebrew install or `nils-cli-pin.yaml`. When you are
done, nothing needs cleanup beyond the helper's cache (see Caching).

## Workflow B — Coupled development against unreleased nils-cli (common loop)

Goal: you opened an issue here, changed `nils-cli`, and came back to adjust
consumers (skill bodies, manifests, renders, fixtures) and test them against the
**unreleased** binary — before any release or bump.

Keep the `nils-cli` work in its own checkout (see `DEVELOPMENT.md` →
"Coupled nils-cli Work"). Then point the helper at it:

```bash
# Use the current nils-cli checkout's debug build (built if missing):
scripts/dev/with-nils-version.sh local -- agent-runtime render --product codex

# Or a specific branch/ref/sha you are iterating on:
scripts/dev/with-nils-version.sh src:my-feature-branch -- bash tests/hooks/run.sh

# Or binaries you already built somewhere:
scripts/dev/with-nils-version.sh path:/abs/path/to/target/debug -- agent-runtime audit-drift
```

Iterate the consumer changes here until the **content** gates (see the gate
section above) pass against the unreleased binary. Two guardrails:

- **Do not commit golden churn produced by the off-pin binary.** If the
  unreleased surface changes rendered output, the golden diff belongs to the
  later `meta:nils-cli-bump` PR (after release), not this branch. On this branch
  the goldens must still match the pinned binary.
- **Do not move the pin here.** This branch ships consumer logic that is
  forward-compatible; the pin moves in lock-step with the release in Workflow C.

When `nils-cli` cuts the release containing your change, proceed to Workflow C.

## Workflow C — Promote the pin (upgrade / formal bump)

Goal: a `nils-cli` release shipped; move the host and the pin to it.

This is owned by the `meta:nils-cli-bump` skill — do not hand-roll it here:

1. `brew upgrade sympoies/tap/nils-cli` so the host is on the target tag (the
   gate is exact, so the host must match before the pin can move).
2. Run `meta:nils-cli-bump`. It diffs the surface, updates
   `docs/source/nils-cli-pin.yaml` and `docs/source/nils-cli-surface.md`,
   refreshes affected consumers/goldens/fixtures, and delivers one bump PR.
3. `scripts/ci/all.sh` Position 2 now passes aligned, and the full stack runs.

The version-switch helper is **not** part of this workflow — Workflow C uses the
real Homebrew release, not a scoped binary.

## The helper: scripts/dev/with-nils-version.sh

A thin wrapper that resolves a version spec to a directory holding the full
`nils-cli` surface, prepends it to `PATH`, prints the resolved
`agent-runtime --version`, and execs your command:

```text
scripts/dev/with-nils-version.sh <spec> [-- <command> [args...]]

<spec>:
  release:<tag>   Download released binaries from the GitHub release page.
  <tag>           Shorthand for release:<tag> when it looks like vX.Y.Z.
  src:<ref>       Build the whole workspace from <ref> (tag/branch/sha) in the
                  nils-cli checkout, via a dedicated worktree.
  local           Use the nils-cli checkout's target/debug build (build if missing).
  path:<dir>      Use binaries already present in <dir>.
```

Environment overrides:

- `NILS_CLI_REPO` — nils-cli checkout for `src:` / `local`
  (default `$HOME/Project/sympoies/nils-cli`).
- `NILS_REPO_SLUG` — GitHub `owner/repo` for `release:` downloads
  (default `sympoies/nils-cli`).
- `NILS_RELEASE_ASSET_PATTERN` — `gh release download --pattern` glob when the
  default platform auto-pick does not match the release's asset names.
- `NILS_BUILD_ARGS` — extra `cargo build` args for `src:` / `local`
  (default builds the whole workspace, debug profile).

### Caching

Downloaded and extracted release surfaces are cached under
`${CLAUDE_KIT_STATE_HOME:-${XDG_STATE_HOME:-$HOME/.local/state}/agent-runtime-kit}/out/nils-versions/<tag>/`
so a repeated `release:<tag>` reuses the extraction. `src:` builds persist in the
nils-cli worktree's `target/`. `local` / `path:` are not cached. Remove the
cache dir to force a fresh download.

## Out of scope

- A versioned Homebrew formula (`nils-cli@x.y.z` on `sympoies/tap`) — the
  release-page download in Workflow A already covers running an older release.
- A `--skip` flag or override for the Position 2 version-alignment gate — the
  gate is intentionally exact; coupled dev runs content gates instead (see the
  gate section), and the pin moves only via Workflow C.
