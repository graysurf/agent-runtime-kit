# Plan: Plan Archive Search Layer — Phase 1 (Catalog)

## Overview

Ship the discovery layer for the `plan-archive` system: a derived,
deterministic, committed `catalog.json` at the archive root, generated by a new
`plan-archive catalog` command from the authoritative `plans/**/metadata.yaml`
and `_index/` snapshots. This closes the gap where an agent that hits a problem
cannot find the relevant past plan/issue without already knowing its ref number,
and it makes `ref → plans` traversal answerable.

Phase 1 also locks the `CatalogRecord` data contract and a single shared
derivation module so Phase 2 (SQLite + FTS5 search) can consume the same records
with **no refactor**. Phase 2 is out of scope here (separate plan).

The cross-repo discovery cue (when/how to consult the archive) is delivered as a
short global-scope `agent-docs` pointer so it is inherited by every repo's
`project-dev` preflight.

## Scope

- In scope:
  - `CatalogRecord` / `CatalogRef` types with deterministic serialization.
  - A shared `catalog` derivation module (the single Phase 1 + Phase 2 source).
  - `plan-archive catalog` command: generate/write/print/filter.
  - Deterministic committed `<archive>/catalog.json`.
  - Regeneration wired into `migrate --apply` and the post-`refresh` flow.
  - `ref → plans` answerable via the catalog (`catalog --refs-to`).
  - Short pointer doc + global `AGENT_DOCS.toml` registration + query-skill cue.
- Out of scope:
  - SQLite index, FTS5, `query --text`, `query --refs-to` indexed mode (Phase 2).
  - Any change to the `_index/` append-only / scrub / refresh fetch path.
  - `files[]` enrichment (PR changed-files); the field is reserved only.
  - Backfilling date prefixes or new source-repo migration (master plans own it).

## Assumptions

1. `metadata.yaml` continues to carry `source.*`, `original_path`, and
   `refs.{issue,pr,mr}`; the catalog reads these as the plan's identity.
2. The latest `_index/` snapshot per ref is the lexically-last `*.json` in the
   ref folder (colon-free ISO8601 filenames), matching `query`'s existing rule.
3. Snapshot JSON exposes `data.title` and `data.state` for the ref title/state
   fields; missing fields degrade to empty, not error.
4. `AGENT_DOCS_HOME` points at this `agent-runtime-kit` checkout in the user's
   environment, so a global-scope entry is inherited across repos.

## Sprint 1: Catalog Contract, Derivation, and Command (nils-cli)

**Goal**: Land the record contract, the shared derivation module, and the
`plan-archive catalog` command with deterministic output; prove `ref → plans`
and grep both resolve over the live archive.

**PR grouping intent**: group
**Execution Profile**: serial

### Task 1.1: Define CatalogRecord contract and shared derivation module

- **Location**:
  - `crates/plan-archive/src/catalog/mod.rs` (new)
  - `crates/plan-archive/src/lib.rs`
- **Description**: Add `CatalogRecord` and nested `CatalogRef` per the
  discussion source's Data Contract (`slug, host, org, repo, date,
  original_path, archive_commit, title, summary, area[], refs[], files[]`;
  each ref carries `url, kind, number, state, title, latest_snapshot,
  fetched_at`). Implement a `catalog` module that scans
  `plans/<host>/<org>/<repo>/<slug>/metadata.yaml`, resolves each ref's latest
  `_index/` snapshot (reuse `query`'s latest-snapshot logic), and reads
  `data.title`/`data.state`. Serialization MUST be deterministic: records sorted
  by `(host, org, repo, date, slug)`, fixed key order, stable formatting. This
  module is the single derivation point Phase 2 reuses unchanged.
- **Dependencies**:
  - none
- **Complexity**: 4
- **Acceptance criteria**:
  - `CatalogRecord`/`CatalogRef` defined with deterministic serde output.
  - Derivation produces one record per archived plan with refs resolved to
    their latest snapshot; plans with no snapshots or no refs degrade cleanly.
  - Re-running derivation on unchanged input yields byte-identical JSON.
  - `files[]` exists in the type but is always empty in Phase 1.
- **Validation**:
  - `cargo test -p nils-plan-archive` (unit: record assembly incl.
    missing-snapshot / no-refs cases; deterministic serialization).

### Task 1.2: Add the `plan-archive catalog` command

- **Location**:
  - `crates/plan-archive/src/cli.rs`
  - `crates/plan-archive/src/catalog/mod.rs`
- **Description**: Add a `Catalog` subcommand: `--write` writes
  `<archive>/catalog.json` (deterministic); without `--write` it prints to
  stdout honoring `--format json|text`. Add `--grep <substr>` and `--area <x>`
  to filter records in memory, and `--refs-to <ref-url>` to return the plans
  that reference a given ref (linear scan over records). Resolve the archive
  clone path the same way `query` does (`--archive` flag, else machine-local
  config). Emit results through the shared `nils-common` envelope.
- **Dependencies**:
  - Task 1.1
- **Complexity**: 3
- **Acceptance criteria**:
  - `plan-archive catalog --write` writes a deterministic `catalog.json`
    covering every archived plan; second run leaves a clean `git diff`.
  - `catalog --grep`, `--area`, `--refs-to` return correct records on live data.
  - JSON output validates against the crate's envelope contract.
  - Shell completion includes the new subcommand.
- **Validation**:
  - `cargo test -p nils-plan-archive` (filter + refs-to unit tests).
  - Manual: `plan-archive catalog --write` on the live clone, then re-run and
    confirm `git diff` is empty.

### Task 1.3: Regenerate catalog on migrate and refresh

- **Location**:
  - `crates/plan-archive/src/migrate/mod.rs`
  - `crates/plan-archive/src/refresh/mod.rs`
- **Description**: On `migrate --apply`, regenerate `catalog.json` and include it
  in the same signed commit as the migrated plan. In the post-`refresh` flow
  (after snapshot write + scrub-log review), regenerate `catalog.json` so
  `fetched_at`/title/state stay current; do not change refresh's no-auto-commit
  contract — regeneration is part of the same staged change the user reviews.
- **Dependencies**:
  - Task 1.1
  - Task 1.2
- **Complexity**: 3
- **Acceptance criteria**:
  - A `migrate --apply` run leaves `catalog.json` regenerated and committed
    alongside the new plan folder.
  - After a `refresh`, regenerating produces a catalog whose ref `fetched_at`
    matches the new snapshot; no snapshot is overwritten (append-only intact).
  - No change to `_index/` scrub/append-only behavior.
- **Validation**:
  - `cargo test -p nils-plan-archive` (migrate/refresh integration with a temp
    archive fixture).
  - `cargo fmt --all -- --check` and `cargo clippy` clean before commit.

## Sprint 2: Cross-Repo Discovery Cue (agent-runtime-kit)

**Goal**: Make "the archive is queryable, and here is how" discoverable in every
development preflight across repos, with narrow wording.

**PR grouping intent**: group
**Execution Profile**: serial

### Task 2.1: Add the global-scope query pointer doc and register it

- **Location**:
  - `docs/source/plan-archive-query-pointer-v1.md` (new, ~12 lines)
  - `AGENT_DOCS.toml`
- **Description**: Write a short pointer doc: the archive holds past
  plans/issues/PRs; consult it **only** before opening a new plan, or when
  diagnosing a suspected recurring / previously resolved problem; discover via
  grep `<archive>/catalog.json`, fetch context via
  `plan-archive query --ref/--plan/--repo`; staleness shown by `fetched_at`,
  refresh on demand; not every task. Register it in `AGENT_DOCS.toml` as
  `context = "project-dev"`, `scope = "global"`, `required = true`,
  `when = "always"`.
- **Dependencies**:
  - Task 1.2 (the doc references `catalog.json` and `plan-archive catalog`)
- **Complexity**: 2
- **Acceptance criteria**:
  - `agent-docs resolve --context project-dev --strict` run from an unrelated
    repo lists the pointer as an inherited global doc with `status=present`.
  - Wording is narrow (no "every task" guidance) and references both
    `catalog.json` and `plan-archive query`.
  - `rumdl check` passes on the new doc.
- **Validation**:
  - `agent-docs resolve --context project-dev --strict --format checklist`
    (from this repo and from one unrelated repo).
  - `rumdl check docs/source/plan-archive-query-pointer-v1.md AGENT_DOCS.toml`

## Issue Closeout Gate

The tracking issue is complete when:

- Sprint 1 (Tasks 1.1–1.3) and Sprint 2 (Tasks 2.1–2.2) are landed on `main`.
- `plan-archive catalog --write` has been run against the live archive and the
  committed `catalog.json` re-runs to a clean `git diff` (determinism evidence
  posted on the issue).
- `agent-docs resolve --context project-dev --strict` from an unrelated repo
  shows the inherited pointer (`status=present`), posted as evidence.
- `cargo test -p nils-plan-archive`, `cargo fmt --all -- --check`, and
  `cargo clippy` are green; `rumdl check` clean on touched docs.

Phase 2 (SQLite + FTS5 search) is explicitly **not** required for this tracker
to close; it opens its own tracker from the same discussion source.
